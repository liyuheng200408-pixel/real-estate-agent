## 定时任务（按执行时间排序）

### 08:00 每日早间客户跟进询问
**动作：** 发送飞书消息询问是否有新客户要跟进

**发送内容：**
- 询问是否有新客户要跟进
- 附带新客户添加模版

**推送至：** 飞书 Home 频道

**回复规则：**
- 如果用户回复"有"或"有新客户" → 立即发送新客户添加模版
- 模版内容：
  ```
  您好，请填写新客户信息：
  客户姓名：
  客户电话：
  客户等级（S/A/B）：
  客户情况描述（预算/需求/意向小区等）：
  下次回访日期（YYYY-MM-DD）：
  经纪人姓名（选填）：
  经纪人电话（选填）：
  ```
- 用户填写完毕后 → 悠悠自动录入跟进系统

---

### 08:30 每日早报
**动作：** 生成今日待跟进清单

**内容：**
- 列出今日需要跟进的S级客户（按最后跟进时间排序）
- 列出今日需要跟进的A级客户
- 检查今天有无预约带看
- 推送给您

**推送至：** 飞书 Home 频道

**08:00 与 08:30 任务联动说明（重要！）**

两个任务通过 `followup_data_悠悠.json` 数据文件联动，形成完整的客户跟进闭环：

**联动流程：**
```
08:00 悠悠发送询问 → 您回复"有新客户"
         ↓
    悠悠发送添加模版 → 您填写客户信息
         ↓
    悠悠将客户录入系统（写入 followup_data_悠悠.json）
         ↓
08:30 悠悠生成今日待跟进清单（从 followup_data_悠悠.json 读取）
         ↓
    自动包含您08:00录入的新客户！
```

**联动原理：**
| 项目 | 说明 |
|------|------|
| 数据文件 | `/root/.hermes/followup_data_悠悠.json` |
| 08:00 写入 | 用户填写模版后，悠悠将新客户写入数据文件 |
| 08:30 读取 | 生成早报时，从同一数据文件读取待跟进客户 |
| 联动效果 | 08:00录入的客户会自动出现在08:30的早报清单中 |

**数据流向：**
```
用户填写模版
    ↓
悠悠解析并写入 followup_data_悠悠.json
    ↓
08:30早报读取数据 → 生成待跟进清单
    ↓
推送至飞书 Home 频道
```

**关键点：**
- 两个任务独立执行，但共享同一数据源
- 08:00录入的客户会在08:30早报中体现
- 形成"早间询问→数据录入→早报呈现"的完整闭环

### */15 9-22 延迟提醒检查（看房完成询问）
**动作：** 检查是否有需要触发的看房完成询问

**更新日志：**
- 2026-04-25：新增 `update` 命令，支持按客户名更新看房时间
  - 用户说"XX客户改到YY时间看房" → 悠悠自动查找并更新
  - 触发时间自动重新计算（看房时间 + 3小时）

**执行流程：**

```
【创建提醒】

         ↓

         ↓

触发时间 = 9:00 + 3小时 = 12:00
         ↓
悠悠告知您提醒已创建

【检查触发】
每15分钟检查一次（9-22点）
         ↓
到12:00时，发现有小陈的提醒待触发
         ↓
发送消息："老板，您今天9点带小陈看房，现在已经过了3小时，客户看完房了吗？"
         ↓
您回复"看了"
         ↓
悠悠发送标记完成模版

【更新提醒】
您告诉悠悠：小陈改到下午2点看房
         ↓
悠悠执行 python3 delayed_reminder_悠悠.py update 小陈 2026-04-26 14:00

         ↓

         ↓
悠悠告知您更新成功
```

**标记完成模版：**
```

意向度（1=强 2=中 3=弱）：
客户顾虑或需求：
下一步建议：
```

**数据文件：** `/root/.hermes/delayed_reminders_悠悠.json`

**推送至：** 飞书 Home 频道

**新建定时任务时，必须把任务执行的详细执行流程以文本形式写在 HEARTBEAT.md 里。**

包括但不限于：
- 任务动作
- 发送内容/格式
- 推送至哪个频道
- 触发条件（如有）
- 回复规则（如有）
- 与其他任务的联动说明（如有）

这条规则必须严格执行。

## 🔴 定时任务脚本更新同步规则（最高优先级，红字标注）

### 核心原则

> **🔴 当定时任务脚本更新后（包括新增功能、修改逻辑、优化流程），必须立即将更新后的工作流同步写进 HEARTBEAT.md 里的对应流程。**

这是一条**不可撼动的绝对规则**，任何其他规则都不能override这条。

### 具体要求

| 场景 | 要求 |
|------|------|
| 脚本新增功能 | 立即在HEARTBEAT.md对应任务下添加新功能说明 |
| 脚本修改逻辑 | 立即更新HEARTBEAT.md里该任务的执行流程描述 |
| 脚本修复bug | 立即在HEARTBEAT.md备注栏注明已修复的问题 |
| 脚本优化流程 | 立即同步更新HEARTBEAT.md的流程步骤 |

### 同步内容

脚本更新时，必须同步到 HEARTBEAT.md 的内容：

1. **执行动作** — 脚本现在做什么
2. **执行流程** — 脚本的执行步骤（用流程图或步骤说明）
3. **数据文件** — 脚本读写哪些文件
4. **推送内容** — 脚本发送什么消息、什么格式
5. **回复规则** — 脚本如何处理用户回复
6. **联动关系** — 该任务与其他任务的关联（如有）

### 同步时机

> **立即同步，不得延迟。**
>
> 脚本更新完成后，第一件事就是更新 HEARTBEAT.md，而不是等到下次对话时才想起来。

### 示例

**场景：** `delayed_reminder_悠悠.py` 新增了 `update` 命令，支持按客户名更新看房时间

**脚本更新后，HEARTBEAT.md 必须同步更新为：**

```markdown

**更新日志：**

  - 用户说"XX客户改到YY时间看房" → 悠悠自动查找并更新
  - 触发时间自动重新计算（看房时间 + 3小时）

**执行流程：**
```
您告诉悠悠：XX客户改到YY时间看房
         ↓
悠悠执行 python3 delayed_reminder_悠悠.py update XX客户 2026-04-26 14:00
         ↓

更新看房时间：09:00 → 14:00

### 违规警告

> ⚠️ **违反此规则的后果：**
>
> - HEARTBEAT.md 与实际脚本不同步
> - 其他协作者或系统查看HEARTBEAT.md时，无法了解脚本的实际工作方式
> - 可能导致定时任务执行不符合预期
> - 可能导致客户跟进流程出错
>
> **因此：脚本更新 = HEARTBEAT.md更新，两者必须同时完成。**
